# The ALU and Register File

The **Arithmetic Logic Unit (ALU)** and **register file** are the computational heart of every CPU. This lesson digs into their internal structure, traces a real instruction through the datapath, and shows how the 1-bit slice scales to 64-bit operations.

## The ALU: A Combinational Circuit

The ALU is a purely combinational circuit — no flip-flops. Its outputs are always the current function of its inputs, and the result is available after the combinational propagation delay (typically 1–4 ns).

### Typical ALU Operations and Opcodes

| ALU Op Code | Operation | Description |
|-------------|-----------|-------------|
| 000 | ADD | A + B (also used for address calculation) |
| 001 | SUB | A − B (= A + NOT(B) + 1, using two's complement) |
| 010 | AND | A AND B (bitwise) |
| 011 | OR | A OR B (bitwise) |
| 100 | XOR | A XOR B (bitwise) |
| 101 | SLT | 1 if A < B (signed), 0 otherwise |
| 110 | SHL | A << B (shift left logical) |
| 111 | SHR | A >> B (shift right logical) |

The 3-bit ALU opcode is generated by the control unit's **ALU control logic**, which combines the instruction's opcode and function fields.

### Status Flags

After each operation, the ALU writes four 1-bit flags:

| Flag | Name | Set when... |
|------|------|-------------|
| Z | Zero | Result is all zeros |
| N | Negative | MSB of result is 1 |
| C | Carry | Unsigned carry-out from MSB |
| V | Overflow | Signed overflow: carry_in_MSB ≠ carry_out_MSB |

These flags are written to a **condition codes register** (x86) or checked by comparison instructions (RISC-V). Branch instructions like `BEQ` and `BLT` read these flags to decide whether to redirect the PC.

## A 1-bit Full ALU Slice

The fundamental building block is a 1-bit ALU slice that performs ADD, AND, and OR, selected by a 2-bit operation code:

```
Inputs: A, B, CarryIn, Op[1:0]
Outputs: Result, CarryOut

Full adder:
  Sum    = A XOR B XOR CarryIn
  CarryOut = (A AND B) OR (B AND CarryIn) OR (A AND CarryIn)

MUX output:
  Op=00 → Result = A AND B
  Op=01 → Result = A OR  B
  Op=10 → Result = Sum  (ADD)
  Op=11 → Result = Sum with B inverted (SUB when combined with CarryIn=1)
```

**Worked example: slice computation for A=1, B=0, CarryIn=1, Op=10 (ADD)**

```
Sum      = 1 XOR 0 XOR 1 = 0
CarryOut = (1 AND 0) OR (0 AND 1) OR (1 AND 1) = 0 OR 0 OR 1 = 1
Result   = Sum = 0
```

## Subtraction via Two's Complement

The ALU never needs a separate subtractor. To compute A − B:

1. Invert all bits of B (one's complement).
2. Add 1 by setting CarryIn = 1 at the least-significant slice.

```
A − B = A + NOT(B) + 1   (two's complement negation)
```

**Worked: 8-bit, A=9, B=5, compute A−B**

```
A =  00001001  (9)
~B = 11111010  (NOT of 00000101)
Cin = 1

    00001001
  + 11111010
  +        1 (CarryIn to bit 0)
  ----------
  100000100  → carry discarded → 00000100 = 4 ✓
```

## Carry-Lookahead Adder

A 32-bit ripple-carry adder has a 32-stage carry chain — too slow. The **carry-lookahead adder (CLA)** computes all carries simultaneously.

Define:
- **Generate**: `G_i = A_i AND B_i` (this bit position will generate a carry regardless of C_in)
- **Propagate**: `P_i = A_i XOR B_i` (this bit propagates a carry from below)

Then: `C_{i+1} = G_i OR (P_i AND C_i)`

Expanding for a 4-bit group:

```
C1 = G0 + P0·C0
C2 = G1 + P1·G0 + P1·P0·C0
C3 = G2 + P2·G1 + P2·P1·G0 + P2·P1·P0·C0
C4 = G3 + P3·G2 + P3·P2·G1 + P3·P2·P1·G0 + P3·P2·P1·P0·C0
```

All four carries are computed in **2 gate levels** (one AND level for each product, one OR level). The final sum bits (`A XOR B XOR C`) add one more XOR level — total: ~5 gate levels vs 32 for ripple-carry.

A modern 64-bit adder uses two levels of 4-bit CLAs = 16-bit partial CLAs feeding a final CLA, achieving full 64-bit addition in O(log 64) ≈ 6 levels.

## The Register File

### Structure

The **register file** is an array of registers with multiple read and write ports. A typical RISC-style file (used in MIPS, RISC-V):

```
32 registers × 32 bits = 1024 flip-flops

Interface:
  Read port 1:  rs1[4:0] ──▶ data_out_1[31:0]   (combinational)
  Read port 2:  rs2[4:0] ──▶ data_out_2[31:0]   (combinational)
  Write port:   rd[4:0], wr_data[31:0], RegWrite (clocked)
```

### Address Decoding and Reading

The read address (5-bit) feeds a 5-to-32 decoder, which asserts one of 32 word-lines. The selected register's outputs are driven onto the data bus through tristate buffers or a large multiplexer.

**Two read ports** means two decoders and two data buses operate in parallel, allowing the CPU to read both source operands `rs1` and `rs2` in the same clock cycle — critical for single-cycle and pipelined designs.

### Write-Enable Logic

The write port uses a one-hot decoder: only register `rd` has its write-enable asserted. All other registers hold their value. The write happens on the rising clock edge when `RegWrite = 1`.

### Special Registers

| ISA | Register | Value/Role |
|-----|----------|-----------|
| RISC-V | x0 (zero) | Hardwired to 0. Writes are discarded. Reading gives 0. |
| RISC-V | x1 (ra) | Return address for function calls. |
| RISC-V | x2 (sp) | Stack pointer. |
| x86 | RIP | Instruction pointer (separate, not in register file). |
| x86 | RSP | Stack pointer. |
| ARM | LR | Link register (return address). |
| ARM | PC | Program counter (accessible as a general register in ARM32). |

The hardwired-zero register (`x0` in RISC-V) is a clever design: it lets the ISA express "discard this result" (write to x0) and "constant 0" (read x0) without adding special opcodes. For example:

```asm
mv   a0, a1       # equivalent to: add a0, x0, a1 (a0 = 0 + a1)
nop               # equivalent to: add x0, x0, x0 (discard result)
```

## Datapath Walkthrough: `ADD x3, x1, x2`

Trace the full path through the 5-stage pipeline in the **Decode + Execute** stages:

```
Stage 2 (ID/Decode):
  ┌──────────────────────────────────────────────────┐
  │  Instruction[19:15] = rs1 = 00001 (x1)           │
  │  Instruction[24:20] = rs2 = 00010 (x2)           │
  │  Register File: read x1 → data1, read x2 → data2 │
  │  Control signals: RegWrite=1, ALUSrc=0, ALUOp=ADD│
  └──────────────────────────────────────────────────┘
           │ data1, data2, control signals stored in ID/EX register

Stage 3 (EX/Execute):
  ┌──────────────────────────────────────────────────┐
  │  ALU input A = data1 (x1 value)                  │
  │  ALU input B = MUX(data2, immediate, ALUSrc=0)   │
  │             = data2 (x2 value)                   │
  │  ALU operation = ADD                             │
  │  ALU result = x1 + x2                            │
  │  Flags: Z=(result==0), N=(MSB), C=(carry), V=(ovf)│
  └──────────────────────────────────────────────────┘
           │ ALU result, flags, rd stored in EX/MEM register

Stage 5 (WB/Writeback):
  ┌──────────────────────────────────────────────────┐
  │  MUX: RegWrite=1, MemToReg=0 → write ALU result  │
  │  Register File: write ALU result to rd = x3      │
  └──────────────────────────────────────────────────┘
```

## Why Registers Instead of Memory?

Registers are on-chip SRAM flip-flops accessible in a **single clock cycle** (< 1 ns). Main memory (DRAM) takes 50–100 ns — 50–100× slower.

| Storage | Latency | Bandwidth | Capacity |
|---------|---------|-----------|----------|
| Register | < 1 ns (0 extra cycles) | N/A | 32 × 32 bits |
| L1 cache | ~1–2 ns (2–4 cycles) | ~1 TB/s | 32–64 KB |
| L2 cache | ~4–10 ns | ~500 GB/s | 256 KB–1 MB |
| DRAM | ~60 ns | ~50 GB/s | 8–128 GB |

The compiler's **register allocator** decides which variables live in registers vs. on the stack (memory). Allocating a hot variable to a register vs. the stack can save 50+ cycles per access in a tight loop.
